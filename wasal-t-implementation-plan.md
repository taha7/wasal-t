## Phase 4 — Auth & API Gateway
> The single entry point. Owns JWT validation and role-based routing.

- [ ] Express app with middlewares: cookie parser, body parser, CORS (allow rider-web and driver-web origins)
- [ ] `POST /auth/register/rider` — creates user (role=rider) + riders profile row, returns 201
- [ ] `POST /auth/register/driver` — creates user (role=driver) + drivers profile row, returns 201
- [ ] `POST /auth/login` — validates credentials, signs JWT with `{ userId, role }`, sets HttpOnly + SameSite=Strict cookie
- [ ] `POST /auth/logout` — clears the cookie
- [ ] `GET /auth/me` — returns the current user's profile from the JWT
- [ ] JWT middleware: reads cookie, validates, attaches `req.user`
- [ ] Role-gating middleware: `requireRole('rider')` / `requireRole('driver')`
- [ ] Reverse-proxy routes to internal services (`/rides`, `/fares`, `/drivers/location`, `/streams/*`); forward `req.user` via internal headers
- [ ] Health check endpoint at `/health`

## Phase 5 — Ride Service
> Owns the rides table, fare estimation, ride lifecycle, and the Kafka producer for matching requests.

- [ ] `POST /fares` — accepts origin + destination; calculates fare (simple distance × rate for demo); INSERTs ride row with `status='draft'`; returns `{ rideId, fare }`
- [ ] `POST /rides/:rideId` (confirm) — validates ride is in `draft` and belongs to caller; UPDATEs `status='pending'`; publishes to Kafka topic `ride-requests` with `{ rideId, origin, destination }`; returns 202
- [ ] `PATCH /rides/:rideId` (driver accept/decline) — validates caller is a driver; updates `status='matched'` + `driver_id` if accepted; publishes Redis Pub/Sub on `offer-result:{rideId}:{driverId}` (wakes Matching Service); on accept, also publishes `ride:{rideId}:matched` (wakes Notification Service for rider SSE)
- [ ] `DELETE /rides/:rideId` (rider cancel) — updates status to `cancelled`; publishes `ride:{rideId}:cancelled` so Matching Service exits its loop
- [ ] `GET /rides/:rideId` — returns ride detail (for both rider and driver, with role check)

## Phase 6 — Location Service
> Ingests driver pings; answers proximity queries.

- [ ] `POST /drivers/location` — accepts `{ lat, lon }`; resolves driverId from JWT; runs in parallel: `GEOADD drivers:available <lon> <lat> <driverId>` and `SET driver:alive:<driverId> 1 EX 10`
- [ ] `GET /nearby-drivers?lat=&lon=&radiusKm=&limit=` (internal — called by Matching Service) — runs `GEOSEARCH`; filters returned candidates by `EXISTS driver:alive:<id>`; returns the alive subset
- [ ] Optional cleanup: stale-member sweep that ZREMs entries whose alive key has expired (run every 60s)
- [ ] Health check at `/health`

## Phase 7 — Matching Service
> Consumes ride requests from Kafka and runs the async matching loop with Redis lock + Pub/Sub bridge.

- [ ] Kafka consumer subscribed to `ride-requests`; for each message, kick off `matchRide(rideId)`
- [ ] `matchRide(rideId)` async loop:
  - [ ] Track `startTime`; track `radiusKm` (start at 5km)
  - [ ] At each iteration, check for cancellation (subscribe to `ride:{rideId}:cancelled` once; resolve a flag if seen)
  - [ ] If elapsed > 3 min and no expansion done → set `radiusKm = 15`
  - [ ] If elapsed > 5 min → publish `ride:{rideId}:no_drivers`, UPDATE rides `status='failed'`, exit
  - [ ] Call Location Service `GET /nearby-drivers`; iterate candidates
  - [ ] For each driver: try `redis.set('driver-lock:'+driverId, rideId, 'NX', 'EX', 15)`; if fails, skip
  - [ ] On lock acquired: call Notification Service to push offer; subscribe to `offer-result:{rideId}:{driverId}` channel; await Promise with 12s timeout
  - [ ] Resolved with `accepted=true` → return; `accepted=false` or timeout → release lock (or let TTL expire), try next driver
  - [ ] After candidates exhausted → `await sleep(2000)`, repeat
- [ ] In-memory Map of pending offers per worker (`{rideId}:{driverId}` → resolve fn) — resilient via Pub/Sub for cross-instance delivery
- [ ] Health check at `/health`
- [ ] Document the "Pub/Sub fire-and-forget" risk and 12s timeout mitigation in service README

## Phase 8 — Notification Service (Consolidated Real-Time Delivery)
> Holds all SSE connections; subscribes to Redis Pub/Sub; dispatches via strategy pattern.

- [ ] Strategy pattern: `Strategy` interface with `deliver(userId, payload)`; concrete `SseStrategy` for now; `FcmStrategy` / `ApnsStrategy` as TODO classes
- [ ] Connection registry (in-memory Map): `userId → SSE response object`
- [ ] `GET /streams/rides/:rideId` — for riders; verifies caller owns ride; opens SSE; registers; sends heartbeat every 20s
- [ ] `GET /streams/drivers/me/offers` — for drivers; opens SSE; registers; sends heartbeat
- [ ] Internal endpoint `POST /notify/driver/:driverId/offer` — called by Matching Service; resolves connected SSE for driver; writes the offer event
- [ ] Redis Pub/Sub subscriber: on `ride:*:matched` → forward to rider's SSE; on `ride:*:cancelled` → forward; on `ride:*:no_drivers` → forward
- [ ] Cleanup on SSE close: unregister the connection
- [ ] Health check at `/health`

## Phase 9 — Driver Simulator
> A worker that fakes a fleet of drivers pinging location and reacting to offers.

- [ ] CLI flags: `--count=20 --city=san_francisco --acceptRate=0.7`
- [ ] On startup: create N driver users via Auth API, log them in (collect cookies/tokens)
- [ ] Each simulated driver: pick a starting point in a bounding box; on a 4s timer, jitter location and `POST /drivers/location`
- [ ] Each simulated driver: open SSE to `/streams/drivers/me/offers`; on offer event, after a small random delay, accept with probability `acceptRate`
- [ ] Stop signal: SIGINT clears all timers and SSE connections cleanly

## Phase 11 — Driver Web App (Next.js)
> User-facing driver flow.

- [ ] Auth pages: register (vehicle info), login
- [ ] Home page: "Go Online" toggle → starts location pings on a 4s timer + opens SSE to `/streams/drivers/me/offers`

## Risks to Watch
- **Redis Pub/Sub fire-and-forget** → threatens Phase 7 (Matching Service bridge). 12s timeout is the safety net; document this clearly. Production fix = Temporal or Kafka.
- **Trust-the-network internal auth** → threatens Phase 13 (production). Acceptable for demo; must be addressed before any real deployment.
- **Concurrent same-driver matching across Matching Service instances** → threatens Phase 7. Redis distributed lock should handle this; verify with concurrent test in Phase 12.
- **EventSource auth quirks** → threatens Phase 10/11. HttpOnly cookie strategy avoids this entirely; verify SSE works correctly across browsers.
- **Stale Redis geo entries** → threatens Phase 6. Two-key TTL pattern (geoadd + alive key) handles it; cleanup job is optional belt-and-braces.
- **Driver offer race when offer expires while user is tapping Accept** → threatens Phase 7/11. The 12s timeout + lock TTL of 15s gives a 3s buffer; UI countdown should match server timeout.
